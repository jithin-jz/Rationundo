RationUndo · റേഷൻ ഉണ്ടോ?

Track ration shop (Fair Price Shop) stock delivery status for Kerala citizens — check whether this month's commodities have arrived at your shop.

Live site → Get started

Overview

The official portal (epos.kerala.gov.in) is slow and only indexes data by District → Taluk → Shop number. RationUndo solves this with a daily scraper that pulls all shop stock data into PostgreSQL, so every user search is served from the local database with zero external calls at request time and sub-200ms responses.

How it works

The read path and the write path are fully decoupled. Users only ever touch the local database; the slow government portal is only hit by a scheduled background job.

User → FastAPI → PostgreSQL (Supabase) → Response
                       ↑
   GitHub Actions (daily 2 AM IST) → httpx scraper → epos.kerala.gov.in

Features

🔎 Flexible search

By shop number, pincode, or place name with fuzzy autocomplete powered by pg_trgm.

📍 Near me

GPS search returns the nearest shops sorted by distance (km) using a haversine calculation.

🗂️ Browse

Drill down District → Taluk → Shop with per-commodity allocated vs received progress bars.

📲 Installable PWA

Malayalam-first UI with an offline app shell via a service worker.

Architecture

The app follows a layered design so business logic is shared across the JSON API, the HTMX fragment endpoints, and the background workers.

  • api/ — thin routers. routes.py returns JSON; htmx_routes.py returns rendered HTML fragments.
  • services/ — business logic (status derivation, sorting, caching) shared by all entry points.
  • repositories/ — database query helpers, including raw SQL for trigram and geo searches.
  • worker/ — ePOS fetch/parse helpers and IST month-cycle utilities used by the scraper.

Data model

Four tables capture shops, their postal locality, and monthly stock status.

pincodes

Postal master data — pincode, post office, district, optional lat/long. Trigram-indexed on post office name.

ration_shops

ARD number, dealer, district, taluk code (tso_code), location, and coordinates. Linked to a pincode.

shop_stock_status

One row per shop per month cycle (unique on shop_id + month_cycle) with a delivery flag and last-checked timestamp.

stock_items

Per-commodity allocated vs received quantities and arrival date for each status row.

API reference

JSON endpoints under /api. HTMX fragment equivalents live under /htmx. All requests are rate-limited to 120/minute.

Local development

Configure .env from .env.example (set DATABASE_URL), then:

# 1. Start local Postgres (port 5433)
docker compose up -d

# 2. Install deps
pip install -e ".[dev]"

# 3. Create tables
alembic upgrade head

# 4. Seed pincodes + discover shops + link + scrape
python scripts/seed_pincodes.py
python scripts/discover_shops.py
python scripts/fuzzy_match.py
python scripts/scrape_all.py

# 5. Run the server
uvicorn app.main:app --reload --port 8000
Tests: pytest -q
Lint: ruff check .

The scraper

A daily GitHub Actions cron runs scripts/scrape_all.py at 2 AM IST. It:

  • Refreshes only pending/partial shops and skips fully-received ones, since those won't change.
  • Resumes safely after a timeout by skipping shops already checked today.
  • Uses a configurable concurrency cap (SCRAPE_CONCURRENCY) with polite jitter.
  • Prunes stock data older than the last 3 month cycles to keep the DB small.
  • A weekly job refreshes shop coordinates and re-links pincodes.

Deployment

All three services run on free tiers:

  • Render hosts the FastAPI web service (render.yaml).
  • Supabase hosts PostgreSQL; the engine is configured for the transaction pooler (NullPool, disabled prepared statements).
  • GitHub Actions runs the daily scraper and weekly geo refresh.

See DEPLOY.md for the full setup. Remember to run alembic upgrade head against the production DB after schema changes.
